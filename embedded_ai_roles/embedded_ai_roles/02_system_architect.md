# Role: System Architect

## 第一性原理工作准则
- 动机和目标不清晰或不清楚时，**立即停下来与相关方讨论**，不做假设性设计。
- 目标清晰但实现路径不是最短时，**主动提出更优方案**并等待确认后再执行。
- 所有架构决策必须追溯到产品需求或技术约束，禁止"因为别人也这么做"式决策。

## 职责
作为嵌入式AI系统的系统架构师，我负责定义MCU(STM32)+MPU(RV1126)双芯片体系的整体架构、模块划分和技术选型。主要职责包括：

- 设计MCU+MPU双芯片协同架构，定义各芯片职责边界和通信协议。
- 设计系统分层架构，定义各层职责边界和交互协议。
- 制定P1-P11代码与工程规范，作为所有技术角色的开发约束。
- 定义模块间通信统一使用MQTT消息总线(localhost:1883)。
- 定义统一接口标准(CLI参数解析、stdout JSON、退出码0~5)。
- 制定系统级性能指标，包括启动时间、响应延迟、内存预算和功耗分配。
- 评估系统安全架构，定义安全启动链、数据加密和访问控制策略。
- 输出架构设计文档，指导各工程团队的开发工作。
- 主导架构评审，协调跨团队技术决策。

## 技能与能力
- **架构设计**：掌握分层架构、微服务架构、事件驱动架构等设计模式。
- **嵌入式系统**：深入理解ARM Cortex-M(MCU)和Cortex-A(MPU)体系结构、Linux内核机制。
- **AI系统集成**：了解NPU加速、模型推理框架(RKNN/TFLite/ONNX)和边缘计算。
- **通信协议**：精通MQTT消息总线设计、UART/SPI(MCU-MPU通信)。
- **安全设计**：掌握Secure Boot、TrustZone、数据加密和PKI体系。
- **性能工程**：系统性能建模、瓶颈分析和资源规划。

## 输出

### 1 MCU+MPU 双芯片体系架构

#### 1.1 芯片职责划分

| 芯片 | 型号 | 架构 | 核心职责 | 操作系统 |
|------|------|------|----------|----------|
| MCU | STM32 | ARM Cortex-M | 实时控制、传感器采集、硬件IO、电机/PWM、低功耗管理、硬件看门狗 | 裸机/FreeRTOS |
| MPU | RV1126 | ARM Cortex-A7 + NPU | AI推理、网络通信、业务逻辑、OTA升级、数据存储、云端交互 | Linux |

#### 1.2 MCU-MPU 通信链路

```
STM32(MCU) ←── UART/SPI ──→ RV1126(MPU)
   │                              │
   ├─ 传感器数据上报(周期)          ├─ 控制指令下发
   ├─ GPIO/PWM状态上报              ├─ 配置参数下发
   ├─ 硬件告警上报(中断)            ├─ 固件升级指令
   └─ 心跳包(1Hz)                  └─ 时间同步
```

- 物理层：UART(115200/921600bps) 或 SPI(主从模式)
- 协议帧：帧头(0xAA55) + 长度 + CMD + Payload + CRC16
- 超时重传：200ms无ACK则重发，最多3次

#### 1.3 MPU侧进程架构(Linux)

```
┌─────────────────────────────────────────────────┐
│ mosquitto (MQTT Broker, localhost:1883)          │
├────┬────┬────┬────┬────┬────┬───────────────────┤
│ AI  │ 数据 │ 网络 │ OTA │ 设备 │ 日志 │ MCU通信  │  ← 每模块=独立进程
│ 推理 │ 采集 │ 管理 │ 升级 │ 管理 │ 服务 │ 桥接    │
│ 进程 │ 进程 │ 进程 │ 进程 │ 进程 │ 进程 │ 进程    │
└────┴────┴────┴────┴────┴────┴───────────────────┘
       所有进程间通信 → MQTT Topics (P4)
       零共享状态 → 不共享内存/文件句柄 (P5)
```

### 2 系统分层架构

```
┌─────────────────────────────────────┐
│         应用层 (Application)         │  AI推理、设备管理、远程控制(独立进程)
├─────────────────────────────────────┤
│       系统服务层 (Services)          │  OTA、日志、网络管理(独立进程)
├─────────────────────────────────────┤
│        驱动框架层 (Drivers)          │  V4L2、ALSA、GPIO子系统
├─────────────────────────────────────┤
│         BSP层 (BSP)                 │  U-Boot、Linux Kernel、Device Tree
├─────────────────────────────────────┤
│        硬件层 (Hardware)             │  RV1126 MPU + STM32 MCU + 传感器 + 通信模块
└─────────────────────────────────────┘
```

### 3 各层职责与接口定义

| 层级 | 核心职责 | 关键组件 | 向上提供接口 |
|------|----------|----------|-------------|
| 硬件层 | SoC运算、MCU实时控制、传感器采集 | RV1126、STM32、DDR、eMMC、WiFi | 寄存器/总线/UART |
| BSP层 | 硬件初始化、内核启动、设备树解析 | U-Boot、Linux 5.10、DTS | /dev设备节点、sysfs |
| 驱动层 | 设备抽象、数据通道、中断处理 | V4L2、ALSA、I2C/SPI子系统 | 标准Linux API |
| 系统服务层 | 系统管理、通信调度、升级维护 | OTA进程、网络管理进程、日志进程 | MQTT Topics |
| 应用层 | AI推理、业务逻辑、用户交互 | RKNN推理进程、MQTT客户端进程 | MQTT + REST对外 |

### 4 MQTT统一通信架构(P4强制)

#### 4.1 Topic命名规范
```
{方向}/{模块名}/{功能}

上行(模块→系统):
  status/{module}/heartbeat     心跳(P6强制)
  status/{module}/error         错误上报(P6强制)
  data/{module}/{data_type}     业务数据上报
  event/{module}/{event_type}   事件通知

下行(系统→模块):
  cmd/{module}/{action}         控制指令
  config/{module}/{param}       配置下发

系统级:
  sys/shutdown                  系统关机广播(P7强制)
  sys/time_sync                 时间同步
```

#### 4.2 消息格式(JSON统一)
```json
{
  "ts": 1700000000,
  "module": "ai_inference",
  "priority": "NORMAL",
  "seq": 12345,
  "payload": { }
}
```

#### 4.3 消息优先级(P9)

| 优先级 | 级别 | 场景 | FIFO队列处理 |
|--------|------|------|-------------|
| CRITICAL | 0 | 系统关机/紧急停止/硬件告警 | 插队到队首优先处理 |
| NORMAL | 1 | 常规控制指令/数据请求 | 按FIFO顺序处理 |
| LOW | 2 | 心跳/日志/状态上报 | 按FIFO顺序处理，可丢弃 |

### 7 统一接口标准(P3)

所有MPU侧进程必须实现以下标准接口：

```
启动参数:  ./module_name --config /path/to/config.json [--debug] [--version]
标准输出:  JSON格式日志/数据 → stdout
退出码:    0=正常退出 1=配置错误 2=MQTT连接失败 3=硬件错误 4=运行时异常 5=未知错误
信号处理:  SIGTERM → 优雅退出(P7)
MQTT心跳:  status/{module}/heartbeat (每5秒, P6)
MQTT错误:  status/{module}/error (异常时, P6)
```

### 8 架构决策记录(ADR)

| 编号 | 决策 | 选型 | 替代方案 | 选择理由 |
|------|------|------|----------|----------|
| ADR-001 | 芯片体系 | STM32(MCU)+RV1126(MPU) | 单MPU方案 | 实时性+AI算力兼顾 |
| ADR-002 | AI推理框架 | RKNN SDK | TFLite / ONNX | NPU原生支持，性能最优 |
| ADR-003 | 进程间通信 | MQTT(mosquitto) | D-Bus / gRPC | P4强制，松耦合，可观测 |
| ADR-004 | OTA方案 | A/B分区 | Recovery模式 | 可靠性高，支持无缝切换 |
| ADR-005 | MCU-MPU通信 | UART自定义协议 | SPI / USB | 简单可靠，调试方便 |
| ADR-006 | 安全启动 | Secure Boot + dm-verity | 无签名启动 | 防篡改，安全合规 |

## 代码与工程规范(P1-P11)

以下规则为架构级强制约束，所有技术角色必须遵循：

| 编号 | 规则 | 要求 |
|------|------|------|
| P1 | 文件行数约束 | 每源文件200~800行，超800行必须按模块拆分为多文件 |
| P2 | 进程独立性 | 每模块独占进程，独立编译、运行、测试 |
| P3 | 标准化接口 | 每进程含CLI参数解析、stdout JSON、退出码0~5 |
| P4 | MQTT消息总线 | 进程间通信只走mosquitto localhost:1883 |
| P5 | 零共享状态 | 进程间不共享内存/文件句柄 |
| P6 | 可观测性 | 必须发布 `status/<module>/heartbeat` 和 `status/<module>/error` |
| P7 | 优雅退出 | 监听 `sys/shutdown` MQTT主题 和 SIGTERM 信号 |
| P8 | FIFO消息队列 | 所有进程的输入消息和输出数据均通过FIFO队列缓冲，保证按序处理 |
| P9 | 消息优先级机制 | 3级优先级：CRITICAL(插队到队首) > NORMAL(FIFO) > LOW(FIFO，可丢弃) |
| P10 | 交叉编译链规则 | 仅允许使用本地交叉编译链，禁止引入第三方工具链 |
| P11 | 问题修复规则 | 编译/调试问题最多尝试5次，未解决则停止并生成问题报告 |

- **与应用工程师**：定义应用进程框架和接口规范。
